# StackUp

Веб-компаньон для домашнего покера: учёт стеков фишек вместо физических фишек.
Без карт, без онлайн-игры на деньги — организатор создаёт стол, игроки заходят по
QR-коду и вводят только имя, организатор вручную фиксирует rebuy/add-on и
корректировки стека, в конце — кнопка «Завершить игру» показывает итоги.

## 1. Настройка Supabase

1. Создайте проект на [supabase.com](https://supabase.com).
2. Откройте **SQL Editor** → по очереди выполните `0001_init.sql`, `0002_player_status.sql`,
   а затем `0003_poker_table.sql` (каждый — отдельным запросом, по очереди).
   Это создаст таблицы, RLS-политики и все нужные функции, а также добавит
   `rooms` и `players` в Realtime-публикацию.
3. В **Project Settings → API** скопируйте `Project URL` и `anon public` ключ.

## 2. Настройка проекта

```bash
cp .env.example .env
# впишите в .env свои VITE_SUPABASE_URL и VITE_SUPABASE_ANON_KEY

npm install
npm run dev
```

Приложение откроется на `http://localhost:5173`.

## 3. Деплой

Это статичный SPA — после `npm run build` папку `dist` можно отдать любым
статическим хостингом (Vercel, Netlify, Cloudflare Pages и т.д.), сервер не
нужен: фронтенд обращается к Supabase напрямую через REST/Realtime. Не
забудьте задать те же переменные окружения (`VITE_SUPABASE_URL`,
`VITE_SUPABASE_ANON_KEY`) в настройках хостинга.

QR-код в панели организатора кодирует ссылку вида `<ваш-домен>/join/<id>` —
поэтому деплой нужно сделать один раз, до того как звать игроков.

## Как это работает

- **Организатор**: Home → «Создать стол» → получает приватный `host_token`
  (хранится только в localStorage этого браузера) → попадает в `/host/:roomId`
  с QR-кодом и списком игроков.
- **Игрок**: сканирует QR → вводит имя на `/join/:roomId` → видит свой стек на
  `/play/:roomId`, который обновляется в реальном времени.
- **Завершение**: организатор жмёт «Завершить игру» → все участники
  автоматически переходят на `/results/:roomId`.
- **Игрок ушёл раньше времени**: кнопка «✕» на карточке игрока в панели
  организатора фиксирует его текущий стек как финальный результат и убирает
  из списка «за столом» — баланс стола (`sum(result) = 0`) при этом
  продолжает сходиться, а в итогах он всё равно появится.
- **История операций**: сворачиваемый список под столом игроков — все
  входы, rebuy/add-on и ручные корректировки с именем игрока и временем.
- **Светлая/тёмная тема**: переключатель ☀️/🌙 в правом верхнем углу,
  выбор запоминается в localStorage; по умолчанию ориентируется на
  системную тему устройства.

### Этап 1 — настоящий покерный стол (новое)

- **Хост тоже может играть**: на экране лобби есть форма «Сесть за стол» —
  хост получает свой стек и место за столом, не теряя прав администратора.
- **Лобби перед игрой**: после создания стола игра не начинается сразу —
  организатор настраивает блайнды, ждёт игроков, и только после нажатия
  «Начать игру» комната переходит в активную фазу.
- **Визуальный стол**: овальный стол с местами по кругу, баттон дилера,
  значки SB/BB, контейнер банка по центру (сам банк подключим на
  следующем этапе вместе со ставками).
- **Ротация дилера и блайндов**: кнопка «Следующая раздача» двигает
  баттон дилера по кругу и увеличивает номер раздачи; блайнды можно
  менять в любой момент через кнопку «Блайнды».
- **Важное ограничение этого этапа**: реальных ставок, банка и
  автоматического списания блайндов ещё нет — фишками всё ещё управляет
  организатор вручную («Стек» / «Rebuy»), как и раньше. Это будет
  следующим этапом.

Вся бизнес-логика выполняется через Postgres RPC-функции
(`create_room`, `join_room`, `adjust_stack`, `add_rebuy`, `remove_player`, `finish_game`) —
прямые INSERT/UPDATE из браузера запрещены через RLS. `host_token` хранится в
отдельной таблице `room_hosts` без единой select-политики, поэтому он не
протекает ни через обычные запросы, ни через Realtime-подписки на `rooms`.
